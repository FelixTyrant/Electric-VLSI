Improvements

Electric может сличать схемы и устанавливать подобие их топологической реализации. Такая операция иногда называется Layout vs. Schematic (LVS), но так как Electric может сравнивать любые схемы(т.е. и топологию, и схематик) будем называть это Network Consistency Checking (NCC).

Есть небольшие различия между Java Electric NCC и C Electric NCC.

  • Java Electric NCC сначала пыталось установить несоответствия микросхем использую новый алгоритм под названием "Local Paritioning". Позволяющий проводить более четкую диагностику. Пара оффициальных ссылок от разработчиков Gemini algorithm (Ebeling, Carl, "GeminiII: A Second Generation Layout Validation Program", Proceedings of ICCAD 1988, p322-325.
  • Java Electric NCC имеет иерархичческий режим. Сопоставление идет в соответствии с выставленной иерархией, что опять же позволяет делать более четкий анализ.
Java Electric NCC так же более быстродейственный, чем C Electric NCC.

Ограничения

NCC имеет некоторые ограничения
  • NCC не проверяет подсоединения подложки транзисторов. Это потому, что не C ни Java не содержат модели подсоединения подложки транзисора. Фактически, топология транзистора не содержит подртов подсоединения подложки.
  • NCC обрабатывает все резисторы как мальнькие микросхемы. Это желаемое состояние резисторов продиктовано Asynchronous Design Group от Sun. Однако, было бы неплохо, если бы определенному резистору в схематике соответствовал тот же самый резистор в топологии. Тем не мение, пока ни C ни Java не могут точно распознать топологический резистор. Наверно скоро этот баг будет исправлен.
  • Отчет о NCС ошибках приводится в стандартном текстовом режиме. Пока нет интеграции с GUI. Опять же обещяют скоро поправить это упущение.
